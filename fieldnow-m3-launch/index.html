Today the NeoGen pipeline shipped its first AI-generated user story directly into a real FieldNOW codebase as an open pull request, with every gate green and three independent layers of verification confirming the result. The substrate — closure-proof chain

Every iteration walks the same chain. Each stage produces a typed verdict (Hold / Fail / Unknown); a failure at any stage blocks the next. If gates fail, the substrate re-dispatches once with the failing witnesses fed back into the prompt as corrective signal (gate-retry); if gates still fail, the iteration is blocked, not shipped.

Fig 1 — Closure-proof chain. Each stage emits a typed K₃ verdict (Hold / Fail / Unknown). Failing gates trigger a one-shot corrective re-dispatch. The bundle is an Ed25519-signed canonical CBOR receipt carrying the entire witness chain.

Architect feedback UI — the human-in-the-loop surface

The automated pipeline (above) is one of two surfaces the substrate exposes. The other is the architect feedback UI — a browser app that surfaces the verifier's witnesses to a human architect and lets each one be resolved with a single click. Every architect decision opens a typed PR on the spec repo with full provenance (witness id, action, tier, decision timestamp). The substrate never silently absorbs a witness; the architect either resolves it or it stays open.

Fig 3 — Architect feedback UI cold start. Setup panel takes a Lovable prototype source, a NeoGen spec target, an envelope file, and (optionally) a client-docs folder for engagement-mode 3. The run panel (capture · specify · generate-stories · run full cycle) drives the pipeline stages.
Fig 4 — Same UI loaded against the real FieldNOW spec: 83 docs captured (2.4 MB), 15 personas / 25 epics / 124 stories, proof verdict Fail (8 open witnesses surfaced for architect resolution). Real artifact state, not a mockup.
Fig 5 — Architect feedback panel with all eight P3 persona-coverage witnesses visible (one per uncovered persona: Subcontractor, Nav, Flow, Quant, Planner, Local Growth, Retention, Paid Search). Each row offers four typed actions — drop (Tier-2 hygiene) · add story (Tier-3 behavioural) · ask client (Tier-1 cosmetic) · amend envelope (Tier-4 architectural). The action click opens a PR on the spec repo with the witness chain in the body; no decision lives only in someone's memory.

Three-way independent verification

The substrate's Hold claim is verified three independent ways for any PR opened by this pipeline. The three layers cross-check each other; the substrate is not its own only witness.

Fig 2 — Three-way independent verification of the same Hold claim. Local substrate runs the gates before the PR opens; the signed bundle lets a reviewer verify the iteration offline using only the bundle CBOR + signature + the public key; GitHub Actions re-runs the same gates from a clean checkout, with no access to the substrate's claim. Three independent oracles converging on the same verdict.

Formal grounding

The substrate is not a pattern of practice — it's grounded on formal mathematics. The K₃ verdict algebra (Hold / Fail(witness) / Unknown) is formally verified in Lean 4 across 16 theorems, with a separate 12-theorem witness-bearing extension and a proven forget homomorphism between the two layers. The substrate composition direction is grounded on Bianconi's higher-order network theory and the surrounding spectral literature; the Hodge decomposition on the skill simplicial complex gives a measurable account of where verification is doing real work versus circling. The Bayesian update calculus follows Jaynes.

The point of the substrate is not that it shipped one story. The point is that the verdict it published — that the story holds — is independently checkable by anyone with the bundle, the verification key, and a clean checkout of the repo. The substrate publishes claims it does not interpret; the verifiers interpret claims they did not produce. Each layer is correct on its own terms.

What this lets us do next

  • Ship more stories. US-1.01.2 (role assignment), US-1.01.3 (team invites) — every story in the FieldNOW backlog can pass through the same pipeline and produce its own auditable PR with the same witness-chain discipline.
  • Independent verification at scale. Auditors, regulators, or partners can fetch any PR's bundle and verify it without trusting the substrate. The verifier doesn't need our code; it needs the bundle, the signature, and the public key.
  • Forward-reference accounting. Each PR's body surfaces the generated code's deferred-work comments as typed dependency edges, cross-referenced against the story backlog. Forward references that don't match an existing story become candidates for new captures — the substrate makes the backlog visible to itself.

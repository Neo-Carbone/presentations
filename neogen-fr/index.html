Vos clients ne veulent pas soumettre des demandes de fonctionnalités. Ils veulent que leur logiciel s'ajuste à leur entreprise. NeoGen permet à chaque locataire de remodeler votre produit par des requêtes en langage naturel, avec un vérificateur dans le chemin de confiance pour qu'aucune modification ne casse rien. Thèmes, tableaux de bord, règles de workflow, même règles d'autorisation — tout s'exprime en compétences typées que le système valide avant d'appliquer. Le vérificateur est la porte de confiance. L'agent n'est que le proposant.

Le problème des ERP de marché intermédiaire.

Dès que votre clientèle dépasse une poignée de segments, le carnet de fonctionnalités cesse d'être une feuille de route et devient plusieurs.

Demandes de fonctionnalités divergentes

Chaque client veut un workflow différent, un tableau de bord différent, un rapport différent. Coder chaque cas à la main ne se met pas à l'échelle.

« Personnalisable » signifie souvent non sécurisé

Les écrans de configuration atteignent vite un plafond. Au-delà, les clients veulent des plugins, des intégrations, des scripts — et c'est vous qui assumez la casse.

Les agents de codage ne règlent pas le problème

Les agents de codage rendent les développeurs plus rapides à écrire le même code. Ils ne rendent pas les changements pilotés par le client sécuritaires.

Les fuites de données entre locataires sont une mort en conformité

L'isolation entre locataires est difficile à bien faire et facile à briser. Une seule constatation d'audit peut entraîner une année de nettoyage.

Comment NeoGen change l'équation.

Trois engagements. Chacun est vérifiable; chacun se compose avec les autres.

1

Génération pilotée par spécifications

Le produit, c'est la spec. Le code en est généré — migrations, politiques RLS, API, tests. Quand la spec change, le code se redérive. Vos ingénieurs écrivent ce qui est véritablement nouveau; le reste, c'est du templating.

2

Compétences comme interface utilisateur

Les clients remodèlent le produit en invoquant des compétences (skills) — opérations typées, à portée de capacités, demandées en langage naturel. Chaque invocation passe par un vérificateur avant d'être appliquée. Les couches d'engagement familières L1/L2/L3 (thématique, structurel, profond) deviennent des formes de paramètres précises du même gabarit générique.

3

Vérificateur dans le chemin de confiance

Chaque vérification retourne une de trois réponses : Hold (c'est sécuritaire), Fail(témoin) (voici exactement ce qui briserait), Unknown (on ne sait pas — escalade). Pas de réessais silencieux, pas de fausse confiance. C'est ce qui rend le changement piloté par le client réellement sécuritaire.

Pourquoi on l'appelle une boucle Ralph. L'agent propose; le vérificateur décide. Le corps de la boucle est volontairement simple — toute l'ingénierie va dans l'oracle qui juge la sortie de l'agent.

Le pipeline qui le fait.

Six étapes, une direction, avec des arêtes de rétroaction qui ferment la boucle. Le pipeline transforme un prototype de départ en un système de production multi-locataire vérifié et signé — et le maintient en synchronisation pendant que vos clients le remodèlent.

Compétences, selon la profondeur de ce qu'elles touchent.

Une compétence est une opération nommée et typée qu'un client peut invoquer. Les couches d'engagement familières L1 / L2 / L3 demeurent le vocabulaire utilisateur. En dessous, chaque couche est une forme de paramètres du même gabarit générique — et le vérificateur lit les paramètres pour décider quoi vérifier.

L1 · cosmétique

Thèmes & libellés

« Mets ma couleur de marque en bleu. » « Modifie le texte du courriel de rappel. » Vérification de types seulement. S'applique immédiatement. Pas d'exposition aux invariants.

L2 · structurel

Tableaux de bord & vues

« Ajoute un KPI de revenus à mon tableau de bord. » « Affiche ces colonnes dans la vue client. » Isolation entre locataires assurée. Portée des données confinée à celles du client.

L3 · profond

Comportement & règles

« Envoie un suivi quand le paiement est en retard de 30 jours. » « Achemine les rehaussements de plan-service à l'équipe abonnements. » Gauntlet d'invariants complet. Trace d'audit signée.

Région architecturale

Autorisation & schéma

« Resserre l'accès à la table clients pour que seuls les administrateurs facturation lisent les montants de factures. » Preuves formelles requises. L'intention du client est honorée; la base de données n'est pas touchée directement.

La même opération peut atterrir dans des couches différentes selon ce qui change précisément. Modifier le texte d'un courriel, c'est L1; recâbler le déclenchement de ce courriel, c'est L3. Le vérificateur lit les paramètres — pas la catégorie.

📖 Que veulent dire ces termes d'audit ? (Hooks · RLS · Politique · Permissive · Verdict)

Hooks (React). Fonctions de logique frontale dans src/hooks/ (p. ex. useCustomers.ts). Ce n'est pas un mécanisme de sécurité; c'est un indicateur de taille. Les 455 hooks signifient « ce prototype porte une logique frontale substantielle. »

RLS (Row-Level Security). La fonctionnalité PostgreSQL qui filtre les lignes qu'une requête peut voir selon le contexte. En soi, c'est seulement un interrupteur; le filtrage réel est fait par les politiques RLS.

Politique RLS. Une règle précise attachée à une table. Anatomie : un nom, une table cible, une opération (SELECT / INSERT / UPDATE / DELETE), une clause USING (filtre de lecture), et optionnellement une clause WITH CHECK (filtre d'écriture). Une table peut avoir plusieurs politiques.

Permissive (au sens de l'audit). Une politique dont la clause USING est littéralement true — elle existe, coche la case « RLS configuré ? » mais ne filtre rien. Chaque ligne passe. Dans une application multi-locataires : les requêtes du locataire A renvoient les lignes du locataire B. L'audit en a compté 498 dans FieldNOW, réparties sur 326 tables distinctes.

Verdict<W>. Trois valeurs, pas deux. Hold = vérifié, passé. Fail(W) = vérifié, brisé — avec un témoin W portant la preuve structurée de ce qui est précisément brisé. Unknown = non vérifié. La logique à deux valeurs confond « pas vérifié » avec « passé »; la logique à trois valeurs force l'honnêteté.

Témoin (Witness). La charge utile structurée à l'intérieur de Fail(témoin). Pour l'isolation locataire, le témoin est une liste de chaque triplet contrevenant (table, politique, raison) — des données typées et actionnables, pas une trace d'erreur.

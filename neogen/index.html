The problem mid-tier ERPs run into.

Once your customer base spans more than a handful of segments, the feature backlog stops being one product roadmap and starts being many.

Divergent feature requests

Every customer wants a different workflow, a different dashboard, a different report. Hand-coding each one doesn't scale.

"Customizable" usually means unsafe

Configuration screens reach a ceiling fast. Beyond it, customers want plugins, integrations, scripts — and now you own the breakage.

Coding agents don't fix it

Coding agents make engineers faster at writing the same code. They don't make customer-driven change safe.

Multi-tenant data leakage is a compliance death

Tenant isolation is hard to get right and easy to break. One audit finding can take a year of cleanup.

How NeoGen changes the equation.

Three commitments. Each is verifiable; each composes with the others.

1

Spec-driven generation

The product is the spec. Code is generated from it — migrations, RLS policies, API, tests. When the spec changes, the code re-derives. Your engineers write what's genuinely novel; the rest is templating.

2

Skills as the user surface

Customers reshape the product by invoking skills — typed, capability-scoped operations they request in natural language. Every invocation goes through a verifier before it applies. The familiar L1/L2/L3 engagement layers (theming, structural, deep) are now precise parameter shapes of the same generic skill template.

3

Verifier in the trust path

Every check returns one of three answers: Hold (it's safe), Fail(witness) (here's exactly what would break), Unknown (we don't know — escalate). No silent retries, no false confidence. This is what makes customer-driven change actually safe.

Why we call it a Ralph loop. The agent proposes; the verifier decides. The loop body is dumb on purpose — all the engineering effort goes into the oracle that judges the agent's output.

The pipeline that does it.

Six stages, one direction, with feedback edges that close the loop. The pipeline turns a starting prototype into a verified, signed, multi-tenant production system — and keeps it in sync as your customers reshape it.

Skills, by the depth of what they touch.

A skill is a named, typed operation a customer can invoke. The familiar L1 / L2 / L3 engagement layers stay as the user vocabulary. Underneath, each layer is a parameter shape of the same generic template — and the verifier reads the parameters to decide what to check.

L1 · cosmetic

Theming & copy

"Set my brand colour to blue." "Edit the reminder email's wording." Type-checked only. Applies immediately. No invariant exposure.

L2 · structural

Dashboards & views

"Add a revenue KPI to my dashboard." "Show these columns in the customer view." Tenant isolation enforced. Data scope confined to the customer's own.

L3 · deep

Behaviour & rules

"Send a follow-up when payment is 30 days late." "Route service-plan upgrades to the membership team." Full invariant gauntlet. Signed audit record.

Architectural region

Authorization & schema

"Tighten access on customers so only billing admins read invoice amounts." Formal proofs gate the change. The customer's intent is honoured; the database isn't touched directly.

The same kind of operation can land in different layers depending on what part of it changes. Editing an email's wording is L1; rewiring when the email fires is L3. The verifier reads the parameters — not the category.

Upstream of the catalogue

Five decisions a human makes — once, signed, before the catalogue exists.

L1 / L2 / L3 skills are derived mechanically from the captured prototype. But they're derived for some target stack — and the prototype rarely pins that down. The Architectural region is where a human, advised by the architect agent, makes a small set of upstream decisions that parameterise everything below. The result is a typed, signed value: the Architectural Envelope.

Language

Rust · TypeScript · Go · …

Team fluency, latency budget, existing prod stack. Pin the language and the entire catalogue retypes against it.

Runtime

Docker · Kubernetes · Serverless

Ops maturity, tenancy model, cost ceiling. Determines what the deploy skill (M4) actually does.

Execution

Sync · Workflow · Event-driven

Long-running workflows? Plain RPC? Event bus? Sets which sync/async L3 skills are even meaningful.

Form factor

Server · Mobile · Edge · Embedded

Where the system runs. Switching to embedded no_std invalidates ~80% of typical L1/L2 skills — surfaced before the catalogue is built.

DevOps

CI · Secrets · Observability · IaC

The operational envelope. Each subdomain a tagged enum, never a free-form string.

Why the envelope is signed and root-hashed into the spec.

Changing Postgres to Cassandra changes the meaning of every L2/L3 skill that touches data. Treating it as a value mutation would silently invalidate the catalogue. Treating it as a new envelope — re-running the architect's decision loop, breaking the spec hash, forcing M3 Generate to refuse — makes the consequences explicit. Architectural changes are a Ralph loop you re-run, not a value you mutate.

The architect-agent interview is itself a Ralph loop one altitude above the skill catalogue: prototype hints + stakeholder answers → propose envelope → adjudicate via verdict → freeze when Hold. Same discipline, applied to the question of what we're even building for.

📖 What do those audit terms mean? (Hooks · RLS · Policy · Permissive · Verdict)

Hooks (React). Front-end UI logic functions in src/hooks/ (e.g. useCustomers.ts). Not a security thing; a sizing signal. The 455 hooks figure says "this prototype carries substantial front-end logic."

RLS (Row-Level Security). The PostgreSQL feature that filters which rows a query can see based on context. By itself just a switch; the actual filtering is done by RLS policies.

RLS policy. One specific rule attached to a table. Anatomy: a name, a target table, an operation (SELECT / INSERT / UPDATE / DELETE), a USING clause (read filter), and optionally a WITH CHECK clause (write filter). One table can have many policies.

Permissive (audit sense). A policy whose USING clause is literally true — it exists, satisfies "is RLS configured?" checkboxes, but filters nothing. Every row passes. In a multi-tenant app: tenant A's queries return tenant B's rows. The audit counts 498 of these in FieldNOW across 326 distinct tables.

Verdict<W>. Three values, not two. Hold = checked, passed. Fail(W) = checked, broken — with witness W carrying structured evidence of what specifically broke. Unknown = not checked. Two-valued logic conflates "didn't check" with "passed"; three-valued logic forces honesty.

Witness. The structured payload inside Fail(witness). For tenant isolation, the witness is a list of every offending (table, policy, reason) triple — actionable typed data, not a stack trace.
